# What still needs correction

## 1. Peak GPU memory is still broken

`0.0 GB` means the measurement is not observing the server-side GPU.

When inference runs through a RunPod API, measuring CUDA memory on your Mac or client process will always produce zero. Memory must be collected inside the GPU-serving process.

Use one of:

* `torch.cuda.max_memory_allocated()`
* `torch.cuda.max_memory_reserved()`
* NVIDIA Management Library
* `nvidia-smi`
* DCGM exporter
* RunPod server-side monitoring

Reset and synchronize around the inference workload:

```python
torch.cuda.reset_peak_memory_stats()
torch.cuda.synchronize()

# Run model inference here

torch.cuda.synchronize()
peak_gb = torch.cuda.max_memory_allocated() / (1024 ** 3)
```

Do not include GPU-memory usage on the resume until this is fixed.

## 2. Define exactly what the F1 score measures

This is critical.

Your confusion matrix appears to measure **diff-level issue detection**:

* Positive: the diff contains an actionable issue.
* Negative: the diff is clean or a hard negative.

If that is correct, do not describe it ambiguously as complete “code-review F1.” That could imply the model correctly identified every exact issue, line, severity, and explanation.

Use terminology such as:

* **Diff-level issue-detection F1**
* **Actionable-finding detection F1**
* **Code-review finding classification F1**

For a stronger benchmark, separately measure:

* Issue category accuracy
* Severity accuracy
* File/line localization
* Exact finding-type match
* Explanation groundedness

## 3. Perfect recall still deserves investigation

All three LLM configurations achieved **1.00 recall**. That may be valid, but it suggests one of two things:

* Positive examples are relatively easy to identify.
* The scoring logic is permissive about what counts as detecting an issue.

Review several outputs to confirm the model is not receiving credit merely for producing any warning on a positive example.

A true positive should ideally require:

* Correct issue category
* Correct finding type
* Correct file or code region
* Explanation supported by the diff

## 4. The benchmark is positively skewed

The benchmark contains:

This is not inherently wrong, but real pull requests may contain a much larger percentage of clean changes.

Consider a secondary benchmark containing more clean examples, such as:

* 50 realistic problematic diffs

This would pressure-test the false-positive rate because noisy review tools fail primarily by commenting on clean code.
